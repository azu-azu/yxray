fetchHistory pattern (new callback, mirrors fetchWatchStatus):
```typescript
const fetchHistory = useCallback(async () => {
  if (!activeProject) return
  try {
    const res = await fetch(
      `/api/history/${activeProject.id}?folder=${encodeURIComponent(activeProject.path)}`
    )
    if (!res.ok) return
    const data = await res.json()
    setHistory(data ?? [])
    setHasCommits((data ?? []).length > 0)
  } catch {
    // ignore network errors — keep stale state
  }
}, [activeProject])
```

Note on hasCommits: After Phase 14, hasCommits is derived from history.length > 0 (set inside fetchHistory). The existing fetchWatchStatus still sets hasCommits too — on project switch, fetchWatchStatus runs first and may set hasCommits=true before fetchHistory populates history. This is acceptable — HistoryPanel renders with empty entries briefly, then fetchHistory populates it. No race condition because both fetch from the same project.

handleUndo after Phase 14: keep existing signature () => void. After undo succeeds in ChangesPanel (which calls the prop), AppShell calls fetchWatchStatus + fetchHistory. SuccessCard no longer needs to call handleUndo — the undo button is now in HistoryPanel. The existing handleUndo in AppShell already calls fetchWatchStatus; extend it to also call fetchHistory.
