# Phase 16.1: Git History UX with Push Integration and Git Graph View - Research

**Researched:** 2026-03-15
**Domain:** React/TypeScript frontend UX extension + Python git backend additions
**Confidence:** HIGH

## Summary

Phase 16.1 enhances the existing HistoryPanel component with three additions that are all well-contained: (1) per-commit push status via cloud icons, (2) a toggleable SVG-based graph view of the commit timeline, and (3) a push button directly in the history panel header. All three build on top of code that already exists and is well-understood: `git_log`, `git_ahead_behind`, `git_push`, the `GET /api/history/{project_id}` endpoint, the `/api/remote/push` endpoint, and the AppShell view-switching mechanism.

The primary technical challenge is the cross-panel communication: after RemotePanel triggers a push, HistoryPanel must refresh its push status without requiring the user to re-open the panel. The CONTEXT.md prescribes using a `lastPushTimestamp` state prop in AppShell (or a Zustand field). The backend work is a new `git_pushed_shas(folder)` function using `git log origin/main --format=%H` and extending the history list response to include `is_pushed: bool` per entry, plus a new `GET /api/remote/push-status` endpoint or an extended history endpoint.

The graph view is intentionally simple for Phase 16.1: a single vertical line with circular nodes (SVG or Tailwind CSS). No third-party graph library is used — the CONTEXT.md explicitly locks this decision. Toggle state persists via `config_store` using a new `history_view_mode` key.

**Primary recommendation:** Build in four sequential waves: (1) backend `git_pushed_shas` + extended history response, (2) cloud icon on EntryRow + push button in HistoryPanel header, (3) SVG GraphView component, (4) AppShell cross-panel push-completion signal. Keep the graph purely visual — no new data fetching in GraphView; it consumes the same `entries` array (now with `is_pushed`) that the list view already uses.

### Pattern 1: Push Status Detection (Backend)

**What:** `git_pushed_shas(folder)` returns the set of commit SHAs that exist at origin/main. The history router then marks each entry with `is_pushed: bool` by checking if its SHA is in that set.

**When to use:** Called during `GET /api/history/{project_id}` response construction.

**Implementation approach:**
```python
# In app/services/git_ops.py
def git_pushed_shas(folder: str) -> set[str]:
    """Return set of commit SHAs that exist at origin/main.

    Returns empty set when no upstream is configured or remote is unreachable.
    Uses git log origin/main --format=%H — lists all commits reachable from origin/main.
    """
    result = subprocess.run(
        ["git", "-C", folder, "log", "origin/main", "--format=%H"],
        capture_output=True,
        text=True,
    )
    if result.returncode != 0:
        return set()
    return set(result.stdout.splitlines())
```

The history router then becomes:
```python
@router.get("/{project_id}")
def list_history(project_id: str, folder: str = Query(...)) -> list[dict]:
    if not git_ops.git_has_commits(folder):
        return []
    entries = git_ops.git_log(folder)
    pushed = git_ops.git_pushed_shas(folder)
    for entry in entries:
        entry["is_pushed"] = entry["sha"] in pushed
    return entries
```

**Why this works:** `git log origin/main --format=%H` lists all commits reachable from origin/main, including commits that may no longer be at HEAD of origin. This is correct — once pushed, a commit stays pushed unless force-push rewrites history (out of scope).

**Edge cases:**
- No upstream configured: `git log origin/main` exits non-zero → returns empty set → all entries get `is_pushed = False`. Correct.
- No remote connected: same result. Cloud icons hidden by frontend check (no remote connected = hide all cloud icons, regardless of is_pushed value).
- Both GitHub and GitLab connected: The `is_pushed` field is a simple bool (pushed to any remote). The tooltip needs per-remote granularity — see Pattern 2.

### Pattern 2: Dual-Remote Push Status for Tooltip

**What:** The cloud icon tooltip shows "Pushed to GitHub · GitLab" or "Pushed to GitHub · Not pushed to GitLab". This requires knowing which remotes each commit has been pushed to.

**Approach:** The simplest implementation is: if `is_pushed=true` AND both remotes are connected, check if both `github_url` and `gitlab_url` exist in config_store for this project. A commit pushed to GitHub was pushed via the `/api/remote/push` with `provider=github` — but the git history doesn't track which provider it went to.

**Practical limitation:** Git doesn't natively track "pushed to GitHub vs GitLab" per commit. The approach from CONTEXT.md is: use `git log origin/main --format=%H` where `origin` is the last-configured remote. This gives "pushed to at least one".

**Recommended approach for tooltip:** Compare `git_pushed_shas(folder)` against the set of commits reachable from each provider's stored remote URL (two separate `git log` calls if both are connected). This requires fetching from each remote — expensive. A simpler approximation: treat "any remote has the commit" as pushed, and surface the partial state by checking if both remotes are connected and whether the project has remote URLs stored for both.

**Practical implementation:** The frontend receives `is_pushed: bool`. For the tooltip, pass `remoteStatus` (already fetched by AppShell/RemotePanel) down to HistoryPanel. If `github_connected && gitlab_connected`, the tooltip text is determined by:
- Both `github_url` and `gitlab_url` in config → show "Pushed to GitHub · GitLab" (all pushed go to both since the push-all button pushes both simultaneously)
- Only one URL configured → only one remote backing, tooltip reflects that

Since the history panel push button "pushes to all simultaneously," once a commit is pushed (via either button), it's pushed to all configured remotes. The tooltip complexity only arises when one remote was connected after some commits were already pushed. For Phase 16.1, implement the simple case: `is_pushed=true` → "Backed up to [list of connected remotes]", partial case handled via tooltip + Go to Remote link.

### Pattern 3: Push Button in HistoryPanel

**What:** Button in header showing `"↑ Push N versions"` where N = ahead count from remote status.

**When to use:** Visible when `remoteStatus.ahead > 0` AND at least one remote is connected.

**Data flow:** HistoryPanel needs `ahead` count and connection status. This means either:
1. HistoryPanel also fetches `/api/remote/status` on mount (self-fetching panels pattern), OR
2. AppShell passes `remoteStatus` as a prop

**Recommended:** Option 1 — HistoryPanel fetches its own remote status. Self-fetching panels is the established pattern. HistoryPanel already receives `projectId` and `projectPath` as props, so it can call `/api/remote/status` on mount.

**Push action from HistoryPanel:** Calls `POST /api/remote/push` for each connected provider simultaneously (or sequentially). After success: re-fetch push status (refresh `is_pushed` flags) and remote status (ahead count → 0).

### Pattern 4: Cross-Panel Push Completion Signal

**What:** After RemotePanel triggers a successful push, HistoryPanel must refresh without user action.

**Implementation:** Add `lastPushTimestamp: number` to AppShell state. RemotePanel calls an `onPushComplete` callback prop after success. AppShell updates `lastPushTimestamp = Date.now()`. HistoryPanel receives `lastPushTimestamp` as a prop and has a `useEffect` that re-fetches remote status + push status when it changes.

```typescript
// AppShell addition
const [lastPushTimestamp, setLastPushTimestamp] = useState(0)

// Passed to HistoryPanel:
<HistoryPanel
  ...
  lastPushTimestamp={lastPushTimestamp}
/>

// Passed to RemotePanel:
<RemotePanel onPushComplete={() => setLastPushTimestamp(Date.now())} />

// In HistoryPanel:
useEffect(() => {
  if (lastPushTimestamp === 0) return
  fetchRemoteStatus()
  // entries re-fetched from parent or via self-fetch
}, [lastPushTimestamp])
```

**Note:** Currently RemotePanel is rendered directly in AppShell's `renderMainContent()` without props. Adding `onPushComplete` requires wiring the RemotePanel component to accept and call this callback after `handlePush` succeeds.

### Pattern 5: SVG Graph View

**What:** A vertical column of circular nodes connected by a vertical line, one node per commit. Identical interaction to EntryRow.

**Approach:** Pure SVG rendered in React. Each node is a `<circle>` with a connecting `<line>` to the next node. The entire SVG is scrollable within the panel.

**Why SVG over DOM:** Clean coordinate math, no layout jank, accessible via role attributes, no additional dependencies.

**Considerations:**
- SVG height = `entries.length * NODE_SPACING`. For long histories, wrap in a scrollable container with `overflow-y: auto`.
- Node click → same `onSelectEntry` callback as list view.
- Cloud icon on node: render a small cloud SVG or foreign object with lucide-react `<Cloud>` component overlaid via absolute positioning on the node.
- Toggle state persisted in `config_store` as `history_view_mode: 'list' | 'graph'` via a `PATCH /api/settings` endpoint (or a dedicated endpoint). Alternatively, use `localStorage` on the frontend since this is a purely UI preference — this is simpler and avoids a new backend endpoint. Use `localStorage.getItem('history_view_mode')` on mount.

**Recommended:** Use `localStorage` for toggle persistence to avoid a new backend endpoint. `config_store` is Python-side; for a UI-only preference, localStorage is the established frontend pattern in this project (already used in the DiffViewer for the localStorage shim).

### Anti-Patterns to Avoid

- **Fetching graph data separately from list data:** GraphView consumes the same `entries` array with `is_pushed` flags. No new endpoint or fetch needed for graph view.
- **Blocking HistoryPanel render on remote status fetch:** Fetch remote status in the background; render list/graph immediately, push button appears once remote status loads.
- **Importing a third-party graph library:** Explicitly locked out by CONTEXT.md.
- **Per-commit git rev-list calls for push status:** `git_pushed_shas` does one `git log origin/main` call; checking each commit individually would be O(n) subprocess calls.

## Common Pitfalls

### Pitfall 1: `git log origin/main` fails when remote is not configured
**What goes wrong:** `git log origin/main --format=%H` exits with code 128 when no upstream is set. If not handled, `git_pushed_shas` raises or returns garbage.
**Why it happens:** No tracking branch exists on fresh repos or before first push.
**How to avoid:** Check `returncode != 0` and return `set()`. The existing `git_ahead_behind` function already demonstrates this pattern — it checks `upstream_result.returncode != 0` and returns `(0, 0)`.
**Warning signs:** Tests pass locally (you've pushed before) but fail in CI on a fresh repo.

### Pitfall 2: Cloud icons shown when no remote is connected
**What goes wrong:** Even if `is_pushed=true` in the API response, cloud icons must be hidden when `remoteStatus` shows neither GitHub nor GitLab connected.
**Why it happens:** `is_pushed` is computed from git state (origin/main exists), but the user may have disconnected in the UI while git remote config still exists.
**How to avoid:** Frontend gate: `if (!remoteStatus?.github_connected && !remoteStatus?.gitlab_connected) — don't render cloud icon regardless of is_pushed`.
**Warning signs:** Cloud icons appearing after user disconnects a remote.

### Pitfall 3: Toggle state lost on project switch
**What goes wrong:** `localStorage` key for `history_view_mode` is global; switching projects should retain the view preference (it's a global preference, not per-project).
**Why it happens:** If keyed per-project, switching projects resets to list view unexpectedly.
**How to avoid:** Store as a single global key `'history_view_mode'`, not per-project.

### Pitfall 4: AppShell does not currently pass `onPushComplete` to RemotePanel
**What goes wrong:** RemotePanel currently accepts no props. Wiring `onPushComplete` requires modifying both the component signature and its call site in AppShell.
**Why it happens:** RemotePanel was built as fully self-contained. Phase 16.1 needs it to signal outward.
**How to avoid:** Add `onPushComplete?: () => void` prop to RemotePanel (optional to avoid breaking the empty-props call site during transition). Call it inside `handlePush` after `setPushState('done')`.

### Pitfall 5: SVG graph height overflow
**What goes wrong:** SVG with fixed height clips nodes for repos with many commits; dynamically sized SVG causes the panel to overflow its container.
**Why it happens:** SVG height must be set explicitly or parent must constrain it.
**How to avoid:** Wrap the SVG in a `div` with `overflow-y: auto` and `flex-1`. Set SVG `height` to `entries.length * NODE_SPACING`. The panel's scroll container already uses `overflow-y: auto` — keep GraphView inside that container.

### Pitfall 6: `is_pushed` field breaks existing test mocks
**What goes wrong:** Existing tests that mock `git_ops.git_log` return entries without `is_pushed`. If the history router now always adds `is_pushed`, tests with old mock return values still pass (router adds the field); but tests asserting exact response shape may fail.
**Why it happens:** Router now mutates entries returned by `git_log` to add `is_pushed`.
**How to avoid:** The router adds `is_pushed` server-side from `git_pushed_shas` return value; existing test mocks for `git_log` don't need updating. New tests for `git_pushed_shas` should mock it separately.

### Pitfall 7: `origin/main` vs `origin/master` branch name
**What goes wrong:** `git log origin/main --format=%H` fails on repos using `master` as default branch.
**Why it happens:** Git default branch name varies; GitHub now defaults to `main`, legacy repos use `master`.
**How to avoid:** Use `git_ahead_behind` approach: get the upstream ref via `git rev-parse --abbrev-ref @{u}` and use that ref for `git log`. Or use `git log @{u} --format=%H` which uses the tracked upstream regardless of name.

```python
def git_pushed_shas(folder: str) -> set[str]:
    # Get the upstream ref name first
    upstream_result = subprocess.run(
        ["git", "-C", folder, "rev-parse", "--abbrev-ref", "@{u}"],
        capture_output=True, text=True,
    )
    if upstream_result.returncode != 0:
        return set()
    upstream = upstream_result.stdout.strip()  # e.g. "origin/main"
    result = subprocess.run(
        ["git", "-C", folder, "log", upstream, "--format=%H"],
        capture_output=True, text=True,
    )
    if result.returncode != 0:
        return set()
    return set(result.stdout.splitlines())
```

### Minimal SVG graph view
```typescript
const NODE_R = 8
const NODE_SPACING = 48
const GRAPH_WIDTH = 40

function GraphView({ entries, onSelectEntry, remoteConnected }: GraphViewProps) {
  const height = entries.length * NODE_SPACING + NODE_R * 2
  return (
    <div className="flex-1 overflow-y-auto p-2">
      <svg width={GRAPH_WIDTH} height={height} className="shrink-0">
        {/* Connecting line */}
        {entries.length > 1 && (
          <line
            x1={GRAPH_WIDTH / 2} y1={NODE_R}
            x2={GRAPH_WIDTH / 2} y2={height - NODE_R}
            stroke="currentColor" strokeWidth={2} className="text-border"
          />
        )}
        {entries.map((entry, i) => {
          const cy = NODE_R + i * NODE_SPACING
          return (
            <g key={entry.sha} onClick={() => onSelectEntry(entry, entry.files_changed[0] ?? '')}
               className="cursor-pointer">
              <circle cx={GRAPH_WIDTH / 2} cy={cy} r={NODE_R}
                fill={entry.is_pushed && remoteConnected ? '#3b82f6' : 'currentColor'}
                className="text-muted-foreground"
              />
            </g>
          )
        })}
      </svg>
    </div>
  )
}
```

## Open Questions

1. **Where does `is_pushed` computation happen when both GitHub and GitLab are connected?**
   - What we know: `git_pushed_shas` uses the upstream ref from `@{u}`, which is one remote. If `git push --set-upstream origin main` was last run for GitHub, `@{u}` points to `origin/main` on the GitHub URL.
   - What's unclear: If the user has pushed to both GitHub and GitLab at different times, `@{u}` always points to the last-pushed remote.
   - Recommendation: For Phase 16.1, `is_pushed=true` means "pushed to at least one remote." The tooltip shows "Backed up" with a generic message or lists connected remotes. The partial-state tooltip ("Go to Remote") is the escape hatch. This is sufficient for Phase 16.1.

2. **Does AppShell re-fetch history entries (with updated `is_pushed`) after a push from HistoryPanel?**
   - What we know: AppShell has `fetchHistory` which fetches `GET /api/history/{id}`. If called after push, the new `is_pushed` values will be computed fresh.
   - Recommendation: After push succeeds in HistoryPanel, call a parent `onPushComplete` callback (which triggers `fetchHistory` in AppShell) — same mechanism as the RemotePanel signal.
