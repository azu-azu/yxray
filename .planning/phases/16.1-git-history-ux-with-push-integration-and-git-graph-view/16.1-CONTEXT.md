# Phase 16.1: Git History UX with Push Integration and Git Graph View - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Enhance the history panel with three additions: (1) push status indicators per saved version showing which commits have been backed up to a remote, (2) a toggleable visual git graph view alongside the existing list view, and (3) a push button directly in the history panel so users can back up without switching to RemotePanel. Branch management and multi-branch graph visualization are Phase 17.

</domain>

<decisions>
## Implementation Decisions

### Push status per entry
- Cloud icon on each history entry that has been pushed to a remote; no icon for local-only entries
- Icons hidden entirely when no remote is connected — no grayed-out state
- Backend: use `git merge-base` + `git log origin/main..HEAD` to identify which commits are local-only vs pushed (N most recent commits matching ahead count are local-only)
- History panel auto-refreshes push status after a push completes in RemotePanel — requires a shared signal/event between panels (not just on panel open)

### Dual-remote push status
- Cloud icon = pushed to **any** connected remote (GitHub OR GitLab counts)
- Hover tooltip on the cloud icon shows exactly which remotes: e.g. `"Pushed to GitHub · GitLab"` or `"Pushed to GitHub · Not pushed to GitLab"`
- Partial state (pushed to one but not both): tooltip includes a `"→ Go to Remote"` link that navigates to RemotePanel for granular per-remote control
- `"Push N versions"` button pushes to **all** connected remotes simultaneously — no per-remote push button in history panel

### Git graph view
- Visual node-and-line commit graph (similar to VS Code git graph extension) — shows commits as nodes connected by vertical lines
- Toggle button in the history panel header switches between list view and graph view: `[ ≡ List | ▦ Graph ]`
- List view remains the default; toggle state persists across sessions via `config_store`
- Clicking a commit node in graph view opens the diff viewer — identical behavior to clicking a row in list view
- Cloud push-status icons also appear on graph nodes (consistent across both views)
- Main branch only — linear single-branch graph; Phase 17 extends to multi-branch DAG

### Push button in history panel
- `"↑ Push N versions"` button appears at the top of the history panel when there are local-only commits (N = ahead count)
- Button is hidden when all commits are pushed or when no remote is connected
- When not connected and user clicks push: navigate to RemotePanel (sidebar view switch) — user sees the connect flow
- After successful push: stay in history panel; push button disappears; previously local-only entries now show cloud icons — no intermediate success message
- Push button appears in both list view and graph view

### Back navigation from diff viewer
- `"← History"` returns to whichever view was active when the user entered the diff viewer (list or graph) — not always list view

### Claude's Discretion
- Exact node styling for the graph view (node size, line thickness, colors)
- Animation/transition when toggling between list and graph view
- Loading state while graph data is fetched
- Error state if push fails from history panel (same pattern as RemotePanel push errors)

</decisions>

<specifics>
## Specific Ideas

- The list/graph toggle sits in the history panel header — compact, doesn't take panel space
- Cloud icon tooltip is the primary way to surface dual-remote push detail — no separate column or badge
- "→ Go to Remote" in the partial-state tooltip is a quick escape hatch, not a call-to-action for normal use
- Graph view is main-branch-only for now — a straight vertical line of nodes, no branching lines until Phase 17

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `app/frontend/src/components/HistoryPanel.tsx`: existing list view — toggle button added to header; `EntryRow` component extended with cloud icon; new `GraphView` component added as sibling
- `app/frontend/src/components/RemotePanel.tsx`: push logic already implemented (`/api/remote/push`); history panel reuses the same endpoint; push completion event needs to propagate to HistoryPanel
- `app/services/git_ops.py`: `git_ahead_behind()` already exists; new `git_pushed_shas(folder)` function needed to return set of commit SHAs that exist at origin (via `git log origin/main --format=%H`)
- `app/routers/history.py`: `GET /api/history/{project_id}` returns commit list — extend response to include `is_pushed: bool` per entry; or add separate `GET /api/history/{project_id}/push-status` endpoint
- `app/services/config_store.py`: use for persisting `history_view_mode` preference ('list' | 'graph') per project or globally

### Established Patterns
- shadcn/ui + Tailwind for all UI — no new component libraries for the graph (build with SVG or CSS, not a third-party graph lib)
- AppShell owns view switching (`activeView` state) — history panel uses `onNavigate('remote')` callback to switch to RemotePanel when push is clicked but not connected
- Self-fetching panels: HistoryPanel fetches its own data on mount — extend to also fetch push status
- Module-level imports in routers for `unittest.mock.patch` compatibility (Phase 11–15 pattern)
- Plain-language copy: "saved versions" not "commits", "backed up" not "pushed" in user-facing text where possible; cloud icon is self-explanatory so no label needed on the icon

### Integration Points
- `app/frontend/src/components/AppShell.tsx`: push completion event — after RemotePanel triggers a successful push, AppShell signals HistoryPanel to re-fetch (e.g. via a `lastPushTimestamp` state prop passed down, or a Zustand store field)
- `app/services/git_ops.py`: new `git_pushed_shas(folder)` function using `git log origin/main --format=%H`
- `app/routers/history.py`: `GET /api/history/{project_id}` response extended with `is_pushed` field per commit entry

</code_context>
