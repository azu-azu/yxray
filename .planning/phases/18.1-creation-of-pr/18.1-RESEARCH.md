## Summary

Phase 18.1 extends the Remote panel in `RemotePanel.tsx` to let users on an experiment branch open a GitHub PR or GitLab MR without leaving the desktop app. Two new backend functions per platform handle creation and existence checks. The `/api/remote/status` response is extended with `pr_exists` and `pr_url` fields so the frontend knows on load whether to show "Open pull request" or "View pull request".

The work splits cleanly into three layers: (1) new service functions in `github_api.py` and `gitlab_api.py`; (2) two new endpoints added to `app/routers/remote.py`; (3) a conditional inline form section added to `renderPushButton()` (or a new helper) inside `RemotePanel.tsx`. All three layers follow patterns already established in Phases 16–17.

The branch name is already available in the Zustand store (`activeBranch[projectId]`) and from the BranchChip component, so no new state infrastructure is required.

**Primary recommendation:** Extend the existing `remote.py` router with `POST /api/remote/pr/create` and `GET /api/remote/pr/status`, extend the `GET /api/remote/status` response with `pr_exists`/`pr_url`, add two functions per API service file, and render the inline form in `renderPushButton` conditioned on `isExperiment && remoteStatus.repo_url`.

### Pattern 1: GitHub API — Create PR

GitHub uses `POST /repos/{owner}/{repo}/pulls`. The owner/repo are derivable from the stored `repo_url` (clone_url contains `https://github.com/{owner}/{repo}.git`). The existing `_github_headers()` function provides auth.

```python
# Source: GitHub REST API docs — https://docs.github.com/en/rest/pulls/pulls#create-a-pull-request
def create_pull_request(
    token: str,
    owner: str,
    repo: str,
    title: str,
    head: str,          # the experiment branch name
    base: str = "main",
    body: str = "",
) -> dict:
    resp = httpx.post(
        f"{GITHUB_API_BASE}/repos/{owner}/{repo}/pulls",
        headers=_github_headers(token),
        json={"title": title, "head": head, "base": base, "body": body},
    )
    resp.raise_for_status()
    return resp.json()  # contains html_url


def get_open_pr_for_branch(token: str, owner: str, repo: str, branch: str) -> dict | None:
    """Return open PR dict for branch, or None if none exists."""
    resp = httpx.get(
        f"{GITHUB_API_BASE}/repos/{owner}/{repo}/pulls",
        headers=_github_headers(token),
        params={"head": f"{owner}:{branch}", "state": "open"},
    )
    resp.raise_for_status()
    items = resp.json()
    return items[0] if items else None
```

**Confidence:** HIGH — GitHub REST API docs verified. `html_url` field on the returned PR object is the web URL. `_github_headers()` already returns the correct Accept and API version headers.

### Pattern 2: GitLab API — Create MR

GitLab uses `POST /projects/{id}/merge_requests`. The project numeric ID is returned in the project creation response stored via `config_store`, but is NOT currently persisted. However, the `repo_url` for GitLab is `http_url_to_repo` which is `https://gitlab.com/{namespace}/{project}.git`. The GitLab project ID can be resolved from the repo URL via `GET /projects/{url-encoded-path}` (namespace/project), or it can be derived from the stored URL.

**Key finding:** GitLab project ID is not stored in `config_store` currently — it only stores the repo URL. The simplest path is to resolve the project ID from the URL each time, or to store it at push time. Two options:

- Option A (recommended): At PR creation time, call `GET /projects/{url_encoded_path}` to resolve the numeric ID from the namespace path. One extra API call, no config change.
- Option B: Extend `config_store.set_remote_repo()` to also persist `gitlab_project_id` when the project is created in the push flow. Requires modifying Phase 16 code.

**Recommendation:** Option A — keeps Phase 18.1 self-contained, avoids touching Phase 16 plumbing.

```python
# Source: GitLab REST API docs — https://docs.gitlab.com/ee/api/merge_requests.html
def _gitlab_headers(token: str) -> dict[str, str]:
    return {"PRIVATE-TOKEN": token}


def get_gitlab_project_id(token: str, namespace_path: str) -> int:
    """Resolve GitLab project numeric ID from 'namespace/project' path."""
    encoded = namespace_path.replace("/", "%2F")
    resp = httpx.get(
        f"{GITLAB_BASE}/projects/{encoded}",
        headers={"PRIVATE-TOKEN": token},
    )
    resp.raise_for_status()
    return resp.json()["id"]


def create_merge_request(
    token: str,
    project_id: int,
    title: str,
    source_branch: str,
    target_branch: str = "main",
    description: str = "",
) -> dict:
    resp = httpx.post(
        f"{GITLAB_BASE}/projects/{project_id}/merge_requests",
        headers={"PRIVATE-TOKEN": token},
        json={
            "title": title,
            "source_branch": source_branch,
            "target_branch": target_branch,
            "description": description,
        },
    )
    resp.raise_for_status()
    return resp.json()  # contains web_url


def get_open_mr_for_branch(token: str, project_id: int, branch: str) -> dict | None:
    resp = httpx.get(
        f"{GITLAB_BASE}/projects/{project_id}/merge_requests",
        headers={"PRIVATE-TOKEN": token},
        params={"source_branch": branch, "state": "opened"},
    )
    resp.raise_for_status()
    items = resp.json()
    return items[0] if items else None
```

**Confidence:** HIGH — GitLab API docs verified. `web_url` field on the MR response is the browser URL.

### Pattern 3: Repo URL Parsing Helper

Both providers need to extract owner/repo or namespace/project from a stored `repo_url`. This is a pure string operation:

```python
# github: https://github.com/owner/repo.git → ("owner", "repo")
# gitlab: https://gitlab.com/namespace/project.git → "namespace/project"

import re

def parse_github_owner_repo(repo_url: str) -> tuple[str, str]:
    """Extract (owner, repo) from GitHub clone URL."""
    m = re.match(r"https://github\.com/([^/]+)/([^/]+?)(?:\.git)?$", repo_url)
    if not m:
        raise ValueError(f"Cannot parse GitHub URL: {repo_url}")
    return m.group(1), m.group(2)


def parse_gitlab_namespace_path(repo_url: str) -> str:
    """Extract 'namespace/project' from GitLab clone URL."""
    m = re.match(r"https://gitlab\.com/(.+?)(?:\.git)?$", repo_url)
    if not m:
        raise ValueError(f"Cannot parse GitLab URL: {repo_url}")
    return m.group(1)
```

These can live in `github_api.py` and `gitlab_api.py` respectively, or as private helpers in the router. Placing in service files is consistent with existing code.

### Pattern 6: Branch Name Humanization

Strip the `experiment/YYYY-MM-DD-` prefix, replace hyphens with spaces, capitalize the first letter. This logic belongs in the frontend (already has access to `activeBranch[projectId]` from Zustand).

```typescript
// Source: CONTEXT.md spec + branch.py _format_branch_name reference
function humanizeBranchName(branch: string): string {
  // Remove "experiment/YYYY-MM-DD-" prefix
  const withoutPrefix = branch.replace(/^experiment\/\d{4}-\d{2}-\d{2}-/, '')
  // Replace hyphens with spaces
  const withSpaces = withoutPrefix.replace(/-/g, ' ')
  // Capitalize first letter only
  return withSpaces.charAt(0).toUpperCase() + withSpaces.slice(1)
}

// Edge cases (Claude's discretion):

// - No date prefix: "experiment/foo-bar" → "Foo bar"  (fallback: strip "experiment/" only)
// - Plain "main": never shown (button hidden for non-experiment branches)
```

### Anti-Patterns to Avoid

- **Fetching branch name from a new API call:** Branch name is already in Zustand — do not add a `/api/branch/current` endpoint. Read `activeBranch[projectId]` directly.
- **Embedding PR logic in the status endpoint:** Keep `GET /api/remote/status` unchanged. Add a separate `GET /api/remote/pr/status` endpoint to avoid slowing down the already-expensive status fetch.
- **Storing GitLab project ID in the URL:** Parse the namespace path from `repo_url` and resolve the numeric ID via API on demand. Do not embed IDs in URLs.
- **Auto-opening browser on PR creation:** CONTEXT.md is explicit — no auto-open. Show link inline only.
- **Module-level service import in routers omitted:** All imports must remain at module level for `unittest.mock.patch` to work. This is a hard project constraint.

## Common Pitfalls

### Pitfall 1: GitLab Project ID Not Stored
**What goes wrong:** GitLab MR creation requires a numeric project ID (`/projects/{id}/merge_requests`), but `config_store` only stores `gitlab_url` (the clone URL string). Calling `/projects/{namespace}%2F{project}` to resolve the ID adds one extra API round-trip.
**Why it happens:** Phase 16 only needed the clone URL for `git push`; numeric ID was not needed then.
**How to avoid:** Add a `get_gitlab_project_id(token, namespace_path)` helper that resolves from stored URL. Do this on demand in the router — do not try to backfill config_store in this phase.
**Warning signs:** 404 errors hitting `/projects/{id}/merge_requests` with a non-numeric ID.

### Pitfall 2: Owner/Repo Parsing Edge Cases
**What goes wrong:** Regex against `https://github.com/owner/repo.git` fails if the URL has no `.git` suffix, or if it's an SSH URL, or if the stored URL has a trailing slash.
**Why it happens:** `create_github_repo()` returns `clone_url` which always includes `.git`, but defensive parsing is still good practice.
**How to avoid:** Make the `.git` suffix optional in the regex (`(?:\.git)?$`). Add test coverage for both `https://github.com/owner/repo.git` and `https://github.com/owner/repo`.
**Warning signs:** `ValueError: Cannot parse GitHub URL` in logs.

### Pitfall 3: PR Creation on Non-Experiment Branch
**What goes wrong:** Button appears even on `main`, and the PR creation fails with "A pull request already exists for main" or creates a meaningless PR.
**Why it happens:** Missing `isExperiment` guard in the frontend render condition.
**How to avoid:** Gate the entire PR section on `isExperiment` (see Pattern 7). Never show the PR button/form when `currentBranch === 'main'`.
**Warning signs:** PR button visible in GitHub tab when on `main` branch.

### Pitfall 4: Duplicate PR Creation
**What goes wrong:** User clicks "Open pull request" twice (slow API), creating two PRs for the same branch.
**Why it happens:** No guard during in-flight request + no server-side duplicate check.
**How to avoid:** (a) Disable the Create button during `prState === 'loading'`. (b) The existing PR check at panel load (`GET /api/remote/pr/status`) means the button won't appear if a PR already exists. The `state = 'open'` filter in the GitHub/GitLab list query provides the server-side guard.
**Warning signs:** Multiple open PRs for the same branch in GitHub.

### Pitfall 5: Module-Level Import Required for Mocking
**What goes wrong:** If new service imports are added inside endpoint functions (lazy import), `unittest.mock.patch("app.routers.remote.github_api.create_pull_request")` won't work — the patch target resolves to the wrong module scope.
**Why it happens:** Project-wide convention (documented in STATE.md: "Module-level service imports in routers for unittest.mock.patch compatibility — Phases 11–18").
**How to avoid:** Ensure `github_api` and `gitlab_api` remain at the top of `remote.py` as module-level imports (they already are via `# noqa: F401`). New functions on those modules are accessible via the existing module references.

### Pitfall 6: Stale PR State After Branch Switch
**What goes wrong:** User is on `experiment/foo`, creates a PR, then switches to `experiment/bar`. The PR link for `foo` still shows in the panel.
**Why it happens:** PR state is stored in component state, not keyed by branch.
**How to avoid:** Reset PR state (`setGithubPRUrl(null)`, `setGithubPRState('idle')`) whenever `activeProjectId` or `activeBranch[projectId]` changes. Add the current branch to the `useEffect` dependency array that drives `fetchStatus()`.

## Open Questions

1. **Where does `currentBranch` come from in RemotePanel during the PR status fetch?**
   - What we know: Zustand `activeBranch[projectId]` has the branch name, set by BranchChip on switch. RemotePanel does not currently read this.
   - What's unclear: If RemotePanel mounts before BranchChip has set the branch (cold load), `activeBranch[projectId]` may be `undefined` → falls back to `'main'`.
   - Recommendation: Read `activeBranch[projectId] ?? 'main'` and include it in the `useEffect` dependency array for `fetchStatus`. Only fetch PR status when `isExperiment` is true (branch starts with `experiment/`).

2. **Should the `/api/remote/pr/status` call be merged into the existing `GET /api/remote/status` response or kept separate?**
   - What we know: CONTEXT.md says "extend status response to include `pr_url` and `pr_exists`". The status endpoint already does a git fetch (slow).
   - What's unclear: Whether adding two API calls (GitHub/GitLab list-PRs) to an already-slow endpoint is acceptable.
   - Recommendation: Keep separate (`GET /api/remote/pr/status?folder=&project_id=&provider=&branch=`), called in parallel with `fetchStatus()` on mount. Cleaner and faster. The planner can choose either approach.

3. **GitLab project ID resolution — one extra API round-trip per PR operation**
   - What we know: The project ID is needed for MR endpoints but only the URL is stored.
   - What's unclear: Whether this is noticeably slow on every panel load.
   - Recommendation: Cache the resolved project ID in component state after first resolution, or store it in config_store at push time. For Phase 18.1 scope, resolve on demand with `get_gitlab_project_id()`.
