# Phase 18.1: Creation of PR - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Users on an experiment branch can open a Pull Request (GitHub) or Merge Request (GitLab) directly from the Remote panel — after pushing, without leaving the desktop app. The feature covers both platforms with full parity. Viewing/merging/reviewing PRs are out of scope.

</domain>

<decisions>
## Implementation Decisions

### Entry Point & Placement
- "Create pull request" / "Create merge request" button lives in the Remote panel (GitHub tab / GitLab tab respectively), surfaced contextually after a push
- In-app form — not a browser handoff. Inline form expands within the Remote panel, same as VS Code GitHub PR extension approach
- Flow: user on experiment branch → pushes → Remote panel shows "Open pull request" button → clicking expands inline form → submit → success state with link

### PR Form Fields
- **Title**: pre-filled from branch name — strip `experiment/` prefix and date, humanize the slug (`experiment/2026-03-15-price-calc-test` → `Price calc test`). User can edit.
- **Description**: optional text area, empty by default. Not required. User can add context or skip.
- **Base branch**: always `main` — no selector. Experiment copies always merge back to main.
- **Required**: title must be non-empty before the Create button is enabled (same pattern as branch creation form in Phase 17).

### Post-Creation Behavior
- Form collapses on success; shows `✓ Pull request created` with a clickable "View on GitHub →" (or "View on GitLab →") link inline. No auto-open browser.
- PR link persists: on future panel loads, if an open PR exists for the current branch, the panel shows "View pull request →" instead of "Open pull request". Fetched fresh via API on panel load.

### GitHub vs GitLab Parity
- Platform-specific labels: GitHub tab uses "Pull request", GitLab tab uses "Merge request". Matches what users see on each platform.
- Full parity: both GitHub and GitLab tabs get the complete create/view flow in this phase.
- Existing PR check: on panel load, check via API if an open PR/MR already exists for the current branch. If yes, skip the form — show "View pull request →" link directly. Prevents duplicates.

### Claude's Discretion
- Exact branch-name-to-title humanization logic (edge cases: single-word slugs, all-numeric names, etc.)
- Loading/spinner state during PR creation API call
- Error message copy for API failures (auth expired, base branch not found, etc.)
- Whether the inline form uses an accordion/collapsible pattern or a card that replaces the button

</decisions>

<specifics>
## Specific Ideas

- VS Code GitHub PR extension is the reference for the in-app form pattern — clean, contextual, no browser needed
- "View on GitHub →" / "View on GitLab →" link after creation — platform-specific wording
- The existing push success pattern (calm inline state, no toast) should be the model for PR creation success

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `app/frontend/src/components/RemotePanel.tsx`: GitHub and GitLab tabs already exist with push/pull controls — PR creation form slots in below the push button on each tab
- `app/services/github_api.py`: `_github_headers()`, `get_github_username()` already available — `create_pull_request()` and `get_open_pr_for_branch()` are new functions to add
- `app/services/gitlab_api.py`: `validate_gitlab_token()` already available — `create_merge_request()` and `get_open_mr_for_branch()` are new functions to add
- `app/routers/remote.py`: existing `/api/remote/*` router — add `POST /api/remote/pr/create` and `GET /api/remote/pr/status` endpoints
- `app/frontend/src/components/BranchChip.tsx`: branch name is accessible in the frontend — use to derive default PR title

### Established Patterns
- Self-fetching Remote panel: fetches `/api/remote/status` on mount — extend status response to include `pr_url` and `pr_exists` for current branch
- Module-level service imports in routers for `unittest.mock.patch` compatibility (Phases 11–18)
- shadcn/ui + Tailwind: `Textarea` component available for description field; `Input` for title
- Inline state pattern for push/pull success — reuse same `'idle' | 'loading' | 'done' | 'error'` state machine for PR creation
- Branch name humanization: strip `experiment/YYYY-MM-DD-` prefix, replace hyphens with spaces, capitalize first word

### Integration Points
- `GET /api/remote/status` response: extend to include `{ pr_url: string | null, pr_exists: bool }` for the active branch + provider
- `POST /api/remote/pr/create`: accepts `{ project_id, folder, provider, title, description? }` — returns `{ pr_url }`
- GitHub API: `POST /repos/{owner}/{repo}/pulls` — title, head (branch), base (`main`), body
- GitLab API: `POST /projects/{id}/merge_requests` — title, source_branch, target_branch (`main`), description
- GitHub existing PR check: `GET /repos/{owner}/{repo}/pulls?head={owner}:{branch}&state=open`
- GitLab existing MR check: `GET /projects/{id}/merge_requests?source_branch={branch}&state=opened`

</code_context>
